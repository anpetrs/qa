# ПУТЬ:

### 1. Пользователь вводит URL или кликает по ссылке

Например: 
```
https://example.com/catalog?id=5
```

### 2. Браузер разбирает адрес:
```
https - протокол;
example.com - домен;
/catalog - путь;
id=5 - query-параметр.
```

### 3. Браузер проверяет, нет ли нужного ресурса локально

Сначала он смотрит, можно ли использовать уже сохранённую версию:
   * HTTP cache браузера;
   * иногда service worker, если сайт его использует;
   * внутренние данные о ранее открытых соединениях.

**Важно**:
кэш страницы и ```/etc/hosts``` - это не одно и то же.
Кэш может хранить сам ресурс или его копию.
```/etc/hosts``` хранит только соответствие имени сайта и IP-адреса.

### 4. Если IP-адрес неизвестен, браузер начинает искать IP-адрес сайта по его доменному имени.

Браузеру нужно понять, куда именно отправлять запрос.
Для этого домен ```example.com``` переводится в IP.

Обычно цепочка такая:
* браузерный DNS cache;
* кэш ОС;
* файл hosts;
* затем запрос к DNS resolver, который уже ищет IP.

### 5. Браузер получает IP-адрес.

Теперь он знает, на какой сервер идти.

### 6. Браузер устанавливает соединение с сервером.

Если это **HTTP**, обычно устанавливается **TCP**-соединение.
Если это **HTTPS**, сначала **TCP**, потом **TLS-handshake**.

#### handshake

  1. браузер запрашивает веб-страницу у сервера по протоколу **HTTPS**,
  2. Сервер высылает копию **SSL-сертификата** с открытым ключом.
  3. **браузер проверяет сертификат**, название которого должно совпадать с именем веб-сайта. Кроме того, проверяется дата действия сертификата и наличие корневого сертификата, выданного надежным центром сертификации.
  4. Если браузер доверяет сертификату -- он генерирует предварительный секрет (**pre-master secret**) сессии на основе открытого ключа, используя максимально высокий уровень шифрования, который поддерживают обе стороны - и **высылает свой публичный ключ серверу**.
  5. **сервер зашифровывает** запрошенную страницу полученным публичным ключём **и отправляет её** браузеру.
  6. защитное соединение утсновлено! 

### 7. Браузер отправляет HTTP-запрос

В запросе есть:
* метод, например GET;
* путь;
* headers;
* иногда cookies;
* иногда body, если это POST или другой метод с телом.

### 8. Запрос попадает на сервер

Но не обязательно сразу в приложение и не обязательно в БД.

Часто цепочка такая:
1. CDN или reverse proxy;
2. load balancer;
3. web server;
4. application server;
5. и только потом, если нужно, запрос к базе данных, кэшу, другим сервисам.

### 9. Сервер обрабатывает запрос

В зависимости от типа ресурса он может:
* отдать готовый статический файл;
* отдать закэшированный ответ;
* собрать HTML динамически;
* сходить в БД;
* обратиться к другим сервисам;
* проверить авторизацию пользователя.

### 10. Сервер отправляет HTTP-ответ

В ответе обычно есть:
* status code, например 200, 301, 404, 500;
* headers;
* body.

  Body может содержать:
  * HTML;
  * JSON;
  * CSS;
  * JS;
  * изображение;
  * другой файл.

### 11.1. Браузер получает HTML и начинает его парсить - начинается РЕНДЕРИНГ

Он читает HTML сверху вниз и строит DOM, Document Object Model.
  DOM - это дерево элементов страницы:
  * html;
  * body;
  * div;
  * p;
  * button и так далее.

### 11.2. Параллельно браузер находит внешние ресурсы

Во время чтения HTML он видит:
* CSS-файлы;
* JavaScript-файлы;
* картинки;
* шрифты;
* иконки;
* видео и прочее.

И начинает загружать их отдельными запросами.

### 12. Браузер загружает CSS и строит CSSOM

CSSOM - это структура со всеми правилами стилей.

Потом DOM и CSSOM будут нужны вместе, чтобы понять:
* как должен выглядеть каждый элемент;
* какие элементы видимы;
* какие размеры, цвета, отступы и шрифты применять.

### 13. Браузер обрабатывает JavaScript

**важно**:
обычный script может остановить парсинг HTML, пока скрипт не загрузится и не выполнится;
```defer``` обычно выполняется после разбора HTML;
```async``` загружается и выполняется независимо.

**JavaScript** может:
* изменять DOM;
* делать новые запросы на сервер;
* менять стили;
* показывать или скрывать элементы.

Из-за этого страница может несколько раз перестраиваться.

### 14. Браузер строит render tree

Это не просто DOM.
В render tree попадают **только те элементы, которые надо рисовать на экране**.

Например, элементы с ``display: none``` туда не войдут.

### 15. layout, reflow

На этом этапе он рассчитывает:
* где находится каждый элемент;
* какая у него ширина и высота;
* как элементы расположены друг относительно друга;
* сколько места они занимают.

### 16. paint

Теперь он отрисовывает:
* текст;
* цвета;
* границы;
* тени;
* изображения;
* фон и другие визуальные части.

### 17. compositing

Если есть слои, например из-за анимаций, transform, fixed-элементов, браузер собирает их в итоговую картинку и показывает пользователю.

## После первой отрисовки работа не заканчивается

Дальше могут происходить:
дозагрузка изображений;
новые API-запросы;
изменение интерфейса через JavaScript;
анимации;
перерисовка части страницы;
обновление данных без полной перезагрузки.

#### Что может мешать рендерингу?

* CSS блокирует отрисовку, пока браузер не поймёт стили.
* JavaScript может блокировать парсинг HTML, если он подключён без async или defer.
* Тяжёлые скрипты и частые изменения DOM могут вызывать повторные layout и paint и замедлять страницу.

Пользователь вводит URL -> браузер проверяет кэш -> если нужно, получает IP через hosts/DNS -> устанавливает TCP/TLS-соединение -> отправляет HTTP-запрос -> сервер возвращает HTTP-ответ -> браузер парсит HTML, загружает CSS/JS -> строит DOM и CSSOM -> формирует render tree -> делает layout -> paint -> показывает страницу.

<summary>СЕТЕВЫЕ ПОРТЫ</summary>

* [Порт](https://www.ittelo.ru/news/chto-takoe-port-v-seti/) — это числовой идентификатор (от 0 до 65535), по которому операционная система понимает, **какому именно приложению отдать входящий пакет**. IP-адрес доставляет трафик до машины, а **порт — до конкретного процесса на ней**. Без этой простой схемы веб-сервер, почтовый домен и база данных на одном хосте не смогли бы работать одновременно.

* Порты – это своего рода "адресные ячейки" на сервере, они направляют потоки данных к нужным приложениям. 

### Примеры использования портов
* **Веб-Серверы**/ Классика: порт **80** (**HTTP**) и **443** (**HTTPS**). Если Вы видите сайт без явного указания порта в адресной строке — браузер подставляет один из них автоматически.
* **Серверы Электронной Почты** / Они принимают и отправляют письма. Изначально протокол SMTP работал только через порт 25. Этот порт до сих пор существует, однако в большинстве пользовательских сетей он заблокирован, так как долгое время использовался спамерами для массовых рассылок. **Порт 587** — это современный стандарт для отправки почты через SMTP, в то время как IMAP обращается к портам **143** или **993** при использовании защищенного соединения.
* **Серверы Баз Данных** / Эти системы используют отдельные порты для обработки запросов. Для **MySQL**, например, характерен порт **3306**.
* **Файловые серверы** / FTP работает на порту **21** (управляющий канал) и **20** (данные). Порт **22** — это SSH, а не SFTP в чистом виде: SFTP — подсистема SSH, и именно поэтому он тоже использует 22-й порт. Сам же FTP без шифрования давно пора заменить на SFTP или SCP — передавать учётные данные открытым текстом в 2026-м несерьёзно.

### Чем отличается порт от IP-адреса?
IP-адрес идентифицирует хост в сети, порт — конкретный процесс на этом хосте. Связка IP:порт называется **сокетом** — это и есть уникальная «точка входа» для сетевого соединения.

<summary>МОДЕЛЬ OSI</summary>

Модель OSI ([Open System Interconnection](https://skillbox.ru/media/code/chto-takoe-model-osi-i-zachem-ona-nuzhna-prepariruem-sloyenyy-pirog-interneta/)), или эталонная модель взаимодействия открытых систем **описывает, как устройства в локальных и глобальных сетях обмениваются данными и что происходит с этими данными**. 

* Слои ничего не знают о том, как устроены другие слои. Это называется абстракцией.
* Самый нижний слой отвечает за **физическое представление данных**, то есть за то, как данные передаются по проводам или с помощью радиоволн, а самый верхний — за то, **как приложения взаимодействуют с сетью**.


### 1-й уровень OSI — физический (L1, physical layer)

* модели OSI данные представляют собой физические объекты — ток, свет или радиоволны. Они передаются по проводам или с помощью беспроводных сигналов.
* Самый известный протокол на физическом уровне — **Ethernet**.

### 2-й уровень OSI — канальный (L2, data link layer)

 * Его задача — проверить целостность полученных данных и исправить ошибки. 
 * канальный уровень умеет кодировать сигналы в биты и передавать их дальше.
 * Устройства канального уровня — коммутаторы и мосты. Они нужны, чтобы передавать фреймы нужному адресату.
 *  данные передаются от одного MAC-адреса к другому.
 *  MAC-адрес устройства, которое отправило нам информацию, нужен, чтобы идентифицировать устройства в локальной сети, состоит из 48 или 64 бит и выглядит примерно так: ```1d:85:25:11:b2:0a```
 * Протоколы канального уровня — [PPP](https://ru.wikipedia.org/wiki/PPP_(%D1%81%D0%B5%D1%82%D0%B5%D0%B2%D0%BE%D0%B9_%D0%BF%D1%80%D0%BE%D1%82%D0%BE%D0%BA%D0%BE%D0%BB)), [CDP](https://ru.wikipedia.org/wiki/CDP).


### 3-й уровень OSI — сетевой (L3, network layer)

* отвечает за маршрутизацию данных внутри сети между компьютерами.
* Маршрутизаторы позволяют разным сетям общаться друг с другом: они используют MAC-адреса, чтобы построить путь от одного устройства к другому.
* Данные на сетевом уровне представляются в виде пакетов. Такие пакеты похожи на фреймы из канального уровня, но используют другие адреса получателя и отправителя — **IP-адреса**.
* Чтобы получить IP-адрес обоих устройств (отправителя и получателя), существует протокол **ARP** ([address resolution protocol](https://ru.wikipedia.org/wiki/ARP)). Он умеет конвертировать MAC- в IP-адрес и наоборот.

### 4-й уровень OSI — транспортный (L4, transport layer)

* на этом уровне происходит передача данных по сети.
* Два главных протокола здесь — **TCP** и **UDP**. Они как раз и отвечают за то, как именно будут передаваться данные.

### 5-й уровень OSI — сеансовый (L5, session layer)

Начиная с этого уровня и выше, данные имеют уже нормальный вид — например, привычных нам JPEG- или MP3-файлов. Задача сети на этих уровнях — представить информацию в понятном для человека виде и сделать так, чтобы пользователь мог её как-то «потрогать».

* Сеансовый уровень управляет соединениями, или сессиями.
* Благодаря сеансам вы можете видеть, что собеседник что-то печатает или отправляет вам картинки или видео. Но главная задача этого соединения — обеспечить стабильное соединение для передачи данных.

### 6-й уровень OSI — уровень представления данных (L6, presentation layer)

* На этом уровне происходит преобразование форматов данных — их кодирование и сжатие. Например, полученные данные могут превратиться в GIF- или MP4-файл.
* То же самое происходит и в обратном порядке: когда пользователь отправляет файл другому человеку, данные сначала конвертируются в биты и сжимаются, а потом уже передаются на транспортный уровень.
* Раз мы собираемся передать данные на другой компьютер, их нужно перевести в бинарный формат. После этого начнётся сам процесс передачи по сети.

### 7-й уровень OSI — прикладной (L7, application layer)

* На нём находятся сетевые службы, которые помогают без проблем сёрфить в интернете.
* Прикладной уровень похож на некий графический интерфейс для всей модели OSI — с его помощью пользователь взаимодействует с другими уровнями, даже не подозревая об этом. Этот интерфейс называется **сетевым**.
* Самые популярные из сетевых интерфейсов — это **HTTP**, **HTTPS**, **FTP** и **SMTP**. А «устройства» здесь — это уже программы: Zoom, Telegram, браузеры.

# IP — Internet Protocol

* не подтверждает доставку пакетов получателю,
* не контролирует целостность данных.
* передача данных происходит без установки соединения.

Главная задача IP — маршрутизация датаграмм, то есть речь идет об **определении пути следования данных по узлам сети**. 

# [TCP — Transmission Control Protocol](https://thecode.media/tcp-ip/)

*  TCP обеспечивает и контролирует передачу данных и следит за надежностью и целостностью.
*  работает поверх IP
*  используется при отправке файлов, почты, при работе веб-серверов: там, где требуется гарантировать корректность переданных данных.

[отличное видео про tcp/ip](https://youtu.be/EJzitviiv2c?si=XlwYL7yNEDiATaeb)

# UDP — User Datagram Protocol

* Обеспечивает передачу данных, не создавая предварительного соединения между ними.
* UDP считают недостаточно надежным, т. к. пакеты могут не только не дойти, но и продублироваться либо прийти не по порядку.
* преимущество: скорость доставки данных. используют для стриминга, звонков, онлайн-игр часто используют UDP;

# [FTP — File Transfer Protocol](https://thecode.media/ftp/)

* для передачи файлов;
* используется при организации удаленного доступа к хостингам (* -сервис, в котором хранятся веб-сайты или веб-приложения и который облегчает к ним доступ*).
* Надежен, гарантирует передачу данных, функционирует по принципу клиент-серверной архитектуры.
* если в процессе передачи возникает какая-то ошибка, то процесс останавливается и выводится сообщение для пользователя. Если ошибок не было, значит, пользователь получил именно тот файл, который нужен, в целости и без недостающих элементов.

*Основная **особенность** FTP в том, что он использует двойное подключение. Одно из них используется для передачи команд серверу и происходит по умолчанию через TCP-порт 21, который может быть изменён. Управляющее соединение существует все время, пока клиент общается с сервером. Канал управления должен быть открыт при передаче данных между машинами. В случае его закрытия передача данных прекращается. Через второе происходит непосредственная передача данных. Оно открывается каждый раз, когда осуществляется передача файла между клиентом и сервером. В случае, если одновременно передаётся несколько файлов, для каждого из них открывается свой канал передачи.*

**FTP client** – это программа, позволяющая подключиться к удаленному серверу по FTP и также выполнять на нем необходимые действия с элементами файловой системы. Клиентом вполне может быть браузер, в адресную строку которого следует ввести адрес, представляющий собой путь до определенного каталога или файла на удаленном сервере, в соответствии с общей блок-схемой URL: ``` ftp://user:pass@address:port/directory/file ```

# SFTP - Secure File Transfer Protocol

– протокол прикладного уровня передачи файлов, работающий поверх безопасного канала. 

* Главное, что отличает SFTP от стандартного FTP и FTPS, это то, что SFTP шифрует абсолютно все команды, имена пользователей, пароли и другую конфиденциальную информацию.
*  SFTP (SSH протокол) использует SSH ключи.

# [SSH — Secure SHell](https://thecode.media/sshhhh/)

* Обеспечивает удаленное управление ОС с применением TCP.
* Схож по функциональности с протоколами **Telnet** и **rlogin**, но, в отличие от них, шифрует весь трафик, при этом сохраняется возможность выбора алгоритма шифрования. Это необходимо для передачи паролей и прочей важной информации.
* SSH дает возможность обрабатывать любые другие протоколы передачи данных.
* SSH позволяет безопасно передавать в незащищённой среде практически любой другой сетевой протокол.
* Для работы по SSH нужен SSH-сервер и SSH-клиент. 

# DNS - Domain Name System

 * система доменных имён (Domain Name System) + протокол, без которого данная система работать бы не смогла.
 * дает возможность клиентским компьютерам запрашивать у DNS-сервера IP-адрес какого-нибудь сайта, плюс он помогает осуществлять обмен БД между серверами DNS.
 * В работе системы также используются протоколы TCP и UDP.

# HTTP - Hypertext Transfer Protocol

- Поначалу это был протокол передачи HTML-документов. Сейчас он задействуется в целях передачи произвольных данных в сети. 
- Stateless -не запоминает, кто отправляет запрос, каждый раз отправляется как в первый раз.
- При обмене информацией протокол HTTP чаще всего использует TCP/IP.
* request состоит из method, URL, headers, body;
* response состоит из status code, headers, body.

# NTP — Network Time Protocol

* для синхронизации локальных часов устройства со временем в глобальной/локальной сети.
* NTP задействует [алгоритм Марзулло](https://habr.com/ru/articles/876536/), в результате чего выбирается наиболее точный источник времени. 
* работает поверх UDP, что позволяет ему достигать максимальной скорости передачи данных. 

 
# [SSL - secure sockets layer / TLS - transport layer security](https://habr.com/ru/companies/1cloud/articles/326292/)

* криптографический протокол для безопасной связи.
* для аутентификации используются асимметричные алгоритмы шифрования (пара открытый — закрытый ключ), а для сохранения конфиденциальности — симметричные (секретный ключ).
* сертификат подтверждает подлинность сервера;
* без установки доверенного сертификата прокси HTTPS нормально не расшифруешь.

[SSL - сертификаты](https://1cloud.ru/services/ssl-globalsign)

### как работает: 
1. сервер в ответ на запрос клиента посылает свой сертификат и параметры шифрования.
2. клиент генерирует мастер-ключ, зашифровывает его открытым ключом сервера и отсылает серверу.
3. Сервер расшифровывает мастер-ключ своим частным ключом и подтверждает свою подлинность клиенту, возвращая ему сообщение, заверенное мастером-ключом клиента.
4. Последующие данные шифруются и заверяются ключами, полученными на основе этого мастера-ключа.

# [SMTP - Simple Mail Transfer Protocol](https://skillbox.ru/media/code/smtp-protokol/)

* важно понимать, что SMTP используется именно для отправки сообщений, а не для их получения.
* для чтения писем применяются протоколы [IMAP](https://en.wikipedia.org/wiki/Internet_Message_Access_Protocol) и [POP](https://en.wikipedia.org/wiki/Post_Office_Protocol).
* В 1995 году протокол получил расширение — **ESMTP (Extended Simple Mail Transfer Protocol)**. Оно сохранило совместимость с базовым SMTP, добавило команды для взаимодействия клиентов и серверов, а также обеспечило поддержку передачи мультимедийных сообщений — с видео, вложениями и картинками. Кроме того, изменённая архитектура ESMTP позволила разработчикам добавлять новые механизмы безопасности.
* SMTP использует **TCP** в качестве транспортного протокола, поэтому в первую очередь он устанавливает TCP-соединение между почтовым клиентом и сервером.
*  [Mailtrap](https://mailtrap.io/) - это специализированная платформа для тестирования отправки почты: она имитирует работу реальных почтовых серверов, но все письма попадают не в настоящий ящик получателя, а в «песочницу», которая доступна только вам.

# COOKIES
фрагменты данных в файле документ, которые сервер отправляет браузеру (приходят с сервера в хедере) для отслеживания действий (анализ) и предпочтений (память) клиентов.
Браузер сохраняет у себя и может отправлять с каждым последующим запросом - так сервер идентифицирует клиента. 
Cам по себе HTTP не может запомнить состояние - для этого нужны куки.
```
пример: Set-Cookie: id=a3fWa; Secure
```
- это открытые данные;
- посылаются в HTTP заголовке к одному и тому же серверу.
- можно указать ограничение по времени, на путь и т.д.
- постоянные куки (удаляются, когда истекает время хранения) -- сессионные куки (удаляются сами, когда закрывается сессия) **!сессии хранятся на сервере!**
- можно контролировать субдомены.
- можно редактировать и отправлять на сервер.
- размер: 4кб
- тип данных: строка

### Атрибуты
- domain=domain.com
- path=/
- expires, max-age
- Secure - от сниффингa
- httpОnly - нельзя отредактировать на клиенте, нет доступа для редактирования в коде страницы - предотвращает доступ к куки через js - атаки типа xss.
- SameSite от csrf-атак - подделка межсайтовых запросов.

# local storage

хранилище данных браузера **без срока давности**
- код в браузере устанавливает, чистит и т.д. данные
- нельзя контролировать субдомены.
- размер: ок. 5-10 мб.
- тип данных: строка

# session storage

хранилище данных в рамках одной сессии (одной вкладки браузера) - т.е. нет доступа между вкладками, даже если один и тот же сайт открыть в соседних вкладках.
- нельзя контролировать субдомены.
- размер: ок. 5 мб.
- тип данных: строка

# кеш

нужен для ускорения получения доступа к информации; 
временно хранит информацию, чтобы ускорить доступ к ней при повторных запосах.
