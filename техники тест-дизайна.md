# Тест-дизайн 

— это процесс разработки техник и методов тестирования. 
**Главная задача тест-дизайна** — подготовить рабочую документацию, то есть разработать сценарии, которые позволят протестировать максимальное количество функций за минимальное время. 

<details>
<summary>ЭКВИВАЛЕТНОЕ РАЗДЕЛЕНИЕ</summary>
.
  
разбиение входных данных на группы, внутри которых **система должна вести себя одинаково**.

**примеры**:

```
Поле «тип файла для загрузки».
Разрешены только: pdf, docx, txt.

Классы:
валидные: pdf, docx, txt;
невалидные: jpg, exe, пустое значение, файл без расширения.
```

```
Пример эквивалентного разделения с календарём:

Есть поле «Дата записи».
По правилам системы записаться можно только на рабочий день. На выходные и праздничные дни запись запрещена.

Эквивалентные классы:
валидный класс: любая дата, которая приходится на обычный рабочий день;
невалидный класс 1: любая дата, которая приходится на субботу или воскресенье;
невалидный класс 2: любая дата, которая приходится на праздник;
невалидный класс 3: пустое значение;
невалидный класс 4: некорректная дата или не дата вообще.
```

<summary>МЕТОД ГРАНИЧНЫХ ЗНАЧЕНИЙ</summary>
.
  
Существует теория, что ошибки часто случаются на границах классов. 

По ISTQB граничные значения проверяют от границ эквивалентных классов. BVA применяется только к упорядоченным классам. Сначала делят входные данные на эквивалентные классы, потом находят их минимумы и максимумы, то есть boundary values. 
В CTFL 4.0 отдельно описаны два варианта: 2-value BVA и 3-value BVA.

**пример**:

```
Поле age допускает значения от 18 до 60 включительно.
Эквивалентные классы:
меньше 18 = невалидно;
18-60 = валидно;
больше 60 = невалидно.
```

* для 2-value BVA достаточно взять 17, 18, 60, 61;
* для 3-value BVA нужно взять 16, 17, 18, 19, 59, 60, 61, 62.

Для создания дерева нужно знать первый шаг и ветвление каждого следующего.

  - Удобно в небольших проверках, когда можно проверить все сценарии, по которым может пройти пользователь или программа.

## PAIRWISE ПОПАРНОЕ ТЕСТИРОВАНИЕ 
- помогает провести наименьшее количество тестов при максимальном покрытии.


Применяется, когда нужно проверить корректность работы системы по нескольким параметрам. Суть метода заключается в том, чтобы каждое значение хотя бы раз побывало в одной точке с другим значением. 

**отвечает на вопрос: как покрыть комбинации экономно, без полного перебора?**

**Как они могут работать вместе:**

Сначала ты строишь **дерево классификаций** и определяешь параметры и значения.
Потом для этих параметров можешь применить **pairwise**, чтобы сократить число комбинаций.
А если в системе есть статусы и сценарии жизни объекта, отдельно применяешь **тестирование состояний и переходов**.

## ТАБЛИЦА ПРИНЯТИЯ РЕШЕНИЙ

используется, когда нужно протестировать систему со множеством параметров и вариантов развития событий. Суть техники в том, что при всевозможных сочетаниях нескольких параметров тестировщик получает разный результат. Важно подобрать нестандартные, редко встречающиеся сочетания, и убедиться, что система корректно их обрабатывает. Как правило, это сочетания, которые не должны случаться, но теоретически могут существовать ― узкие места системы. 

```
На примере кредитного калькулятора выделим параметры, при соблюдении которых банк выдаст кредит: 
●	возраст от 18 до 65 лет;
●	человек работает; 
●	цель понятная — например, автокредит, потребительский или ипотечный. 

Различные комбинации трёх параметров приводят к разному, заранее заложенному  поведению системы.
```

### Разница с другими техниками

* С деревом классификаций:
  
  * дерево просто раскладывает параметры и значения;
  * таблица решений показывает, **какой результат должен быть при их сочетании**.

* С pairwise:

  * pairwise сокращает комбинации;
  * таблица решений обычно не сокращает, а фиксирует именно бизнес-правила и ожидаемый результат.

* Со state transition:

  * state transition про изменение статусов во времени;
  * decision table про правила выбора результата в конкретный момент.


# ИТОГО: 

* если есть много условий и надо понять, какое решение система должна принять, это **decision table**;
* если есть статусы и смена статусов, это **state transition**;
* если есть много параметров и комбинаций, это **classification tree** или **pairwise**.
